# Coins Backend Ledger Foundation Report

## Schema Design

- Added `supabase/coins_ledger_schema.sql`.
- Introduced `coin_accounts` as the backend-authoritative per-player Coins balance table.
- Introduced `coin_ledger_entries` as the immutable balance movement log.
- Added indexes for `player_id`, recent ledger history lookup, and optional `external_ref`.
- Enabled RLS with no public policies so normal client roles cannot directly read or mutate Coins tables.

## Backend Functions

- Added `getCoinBalance` player endpoint.
  - Validates Telegram `initData`.
  - Resolves the current player server-side.
  - Creates a `coin_accounts` row with `0` balance when missing.
  - Returns `balanceCoins` and a limited recent ledger list.
- Added shared `coinsLedger` helper for account creation, balance lookup, credit, debit, and ledger writes.
- Added optional `adminAdjustCoins` endpoint guarded by `ADMIN_SECRET`.

## Frontend Display Changes

- The market/TON panel now shows backend-driven Coins information only.
- Added loading state, error state, `Монети: ...` display, and the message:
  `Купівля монет через TON буде додана наступним етапом.`
- No purchase button, withdrawal flow, chest UI, reroll UI, or fake local Coins balance was added.

## Security Rules

- Coins are not stored in `HeroState` as source of truth.
- Cloud save sanitization strips `coins`, `coinBalance`, `balanceCoins`, and related premium balance fields from client payloads.
- `savePlayerState` and `adminUpdatePlayer` both strip any coin-like fields before persistence.
- Coins are not generated by gameplay, old gold conversion, crafting, inventory, or regular player save sync.
- Negative balances are rejected by backend helper logic unless explicitly allowed for future privileged flows.
